# GEAR FIGHT — 引き継ぎメモ

このファイルは、別環境でこのプロジェクトを開いたときにすぐ状況を把握できるようにするための引き継ぎ資料です。詳細な機能仕様は `docs/仕様書.md`、スプライト生成の詳細は `docs/sprite_generation_prompts.md` を参照してください。

## プロジェクト概要

中学技術科「機械・動力の伝達」学習用の2D対戦格闘ゲーム。`docs/仕様書.md` が元の完全仕様書だが、**以下の点で実装は仕様書と意図的に異なる**：

- 仕様書は「GAS単一HTML・fillRectベースの手続きスプライト」を指定しているが、実装は **Vite + TypeScript + Phaser 3 + Firebase**（デプロイはFirebase Hosting）。理由：既にこの構成で実装が進んでいたため、ゲームプレイ本質（ギア/ヒート/パーフェクトシフト等）に影響しないGAS制約は放棄した。
- スプライトは fillRect ではなく、**実画像（AI生成 + 手続き合成のハイブリッド）** を使用。詳細は下記「スプライト素材」を参照。
- それ以外（ギアシステム、ヒート、パーフェクトシフト、THEORY BONUS、クイズ、AIプロファイル等）は仕様書の記述に準拠する方針。

  - **ドリフトの緑装甲で踏んだ落とし穴**：グリーンバック除去のbg_mask判定（`g>r+30 & g>b+30 & g>100`）が緩すぎて、ドリフト自身の迷彩グリーン（例: RGB(65,113,10)）まで背景と誤判定し透過で消えてしまった（純粋な背景グリーンはRGB(2,234,4)のようにRがほぼ0なのに対し、キャラの緑はR成分が30〜150程度残っている）。対策：背景判定に`r<40`を追加して厳格化。また緑スピル抑制も、前景全体に適用すると同様にキャラの緑を壊すため、`ndimage.binary_dilation`で背景マスクの境界数px以内（エッジリングのみ）に処理範囲を限定。この2つの対策で自身が緑系のキャラでも安全に抽出できる（落とし穴#12の類似ケースとして今後も注意）

- バトルシステムの操作感・爽快感の底上げ（2026-07-11、ストリートファイター2的な手触りを目標に）：
  - **【最重要・致命的バグ修正】通常攻撃が実質発生しなかった不具合**：`Fighter.processInput()`の早期return対象リスト（`hitstun`/`knockdown`/`dead`）に`attack_weak`/`attack_strong`が含まれておらず、攻撃を出した次のフレームで（consumeBufferedが空振りするため）`handleMovement()`まで処理が落ち、`state`が問答無用で`idle`/`walk`に上書きされていた。結果、`tickState()`側の「`elapsed === gear.startup`でヒットボックス発生」に到達する前に毎回攻撃がキャンセルされ、**通常プレイでは弱/強攻撃が(アシストモードでも)ほぼ絶対に当たらない**状態だった（`blocking`分岐にだけ`this.state !== 'attack_weak' && ...`という同種のガードが元々あり、作者は本来この保護を意図していたが、以降の分岐に付け忘れたのが原因と推測）。ヘッドレスブラウザで`scene.events.on('postupdate', ...)`によるフレーム単位トレースを取り、`attackActive`が一度もtrueにならないまま`state`がidleに戻る様子を実際に再現して特定。修正は该当ガードに`attack_weak`/`attack_strong`を追加するだけ（`blocking`分岐側の個別ガードは不要になったので削除）。修正後は同じトレースでヒットボックス発生→ダメージ反映（960/1000、GL3弱攻撃の期待値通り）を確認済み。
  - **ガードの誤動作を修正（ブロックストン導入）**：`Fighter.takeDamage()`がガード成立時も含めて`state`を無条件に`hitstun`/`knockdown`にしていたため、**ガードした瞬間に（防御成功にもかかわらず）操作不能な被弾状態に陥り、ガードを構え続けられなかった**。新しい`FighterState = 'blockstun'`（`BLOCKSTUN_FRAMES=8`、`config/constants.ts`）を追加し、ガード成立時はこちらに遷移。`blockstun`は`hitstun`と同様に一時的に操作不能だが、`processInput()`の早期returnリストに含めることで「ブロックストン明け後、まだ後ろを入力していれば自動的に`block`へ復帰」という、SF2のガード連打が効くのと同じ挙動になる（ガード崩れ時=`guardGauge`0到達時は従来通り本当の`hitstun`/`knockdown`に落ちる）。
  - **ガードゲージが一生回復しなかった不具合を修正**：`guardGauge`は減る一方で回復処理がどこにも無く、一度ガードクラッシュすると以降そのラウンド中ずっとガード無効という設計ミスがあった。`Fighter.tickGuard(deltaSec)`を追加し、`block`/`blockstun`中以外は`GUARD_REGEN_PER_SEC=20`で自然回復するように（`BattleScene.update()`から`tickHeat`と並べて毎フレーム呼び出し）。`BattleHUD`にも小さいガードゲージバー（残量30%未満で赤変化）を追加し、可視化した。
  - **ノックバック追加**：ヒット/ガード時にdefenderがattacker.facing方向へ押し出されるように（`Fighter.applyKnockback()`、`BattleScene.checkHit()`から呼び出し）。通常ヒットで6px、ガードで3px、ダウン時+4px、強攻撃はさらに+4px（`KNOCKBACK_*`定数）。攻撃を当てた時の物理的な手応えが皆無だった（キャラが突っ立ったまま殴り合うだけだった）のを解消。
  - **フレームデータを全ギアで短縮（体感速度アップ）**：`GEAR_TABLE`の`startup`/`recovery`が全ギアで実時間換算0.6〜1.3秒(!)というレベルで、SF2的な軽快さには程遠かった。相対関係（低ギア=速いが弱い、高ギア=遅いが強くガードブレイク可）は維持したまま絶対値を全体的に短縮（例：GL3弱攻撃は startup10/recovery24→7/14）。強攻撃の追加余韻フレームも6→4（`STRONG_ATTACK_EXTRA_FRAMES`）。マジックナンバーだった`hitstun=12`/`knockdown=30`/`invuln=10`/`knockdown閾値=60`も全て名前付き定数化（`HITSTUN_FRAMES`等）。
  - 検証はヘッドレスブラウザ＋`Fighter.bufferInput()`直接注入（CLAUDE.md既存の推奨手法）で実施。①素の弱攻撃がノックバック付きで命中、②ガード中は`block`⇄`blockstun`を繰り返し`hitstun`には絶対に落ちない、③ガードゲージが未ガード時に自然回復、④ミックス入力60ステップのクラッシュ耐性、の4系統を確認。
  - **AIにガード行動を追加**（`AIController.ts`）：以前は`sonica`プロファイル（kakashi以外の全キャラが暫定流用）が一切ガードせず殴られ放題だったのを修正。`AIController.update()`の先頭で`tryBlock()`を呼び、相手（`opponent.state`）が`attack_weak`/`attack_strong`に入ったのを検知すると、①今回のヒットをガードするかどうかを一度だけ確率的に判定（`blockChance = 0.4 + stage*0.15`、`escalateIfLosing()`で上がる`stage`と連動＝劣勢を悟るとAIも堅くなる）、②ガードすると決めた場合のみ3〜7フレームのランダムな反応遅延を挟んでから実際にガード入力を保持する（人間の反応時間に近い感触にするため、フレーム1での超反応=見切りゲーになるのを回避）。この判定・保持ロジックは通常の`thinkTimer`（行動決定のペース）を経由せず毎フレーム走るため、ガード入力を攻撃継続中ずっと保持できる。なお`Fighter.processInput()`は攻撃中の状態遷移を一切受け付けない（前述の致命的バグ修正の一部）ため、AI自身が既に攻撃を振っている最中は原理上ブロックに割り込めない＝振り逃げ/差し合いの駆け引きが自然に生まれる。ヘッドレスブラウザで、①`Math.random`を固定してガード確定/反応遅延後に確実に`block`→`blockstun`→`block`と遷移しチップダメージのみで済むケース、②ガード非確定時は通常通りフルダメージが通るケース、の両方を確認済み。カカシくん（チュートリアル用の的キャラ）は`profile==='kakashi'`のため対象外のまま（意図的、10発当てるチュートリアルの練習台という役割を壊さないため）。
- **超必殺技（必殺技/EX技）の実装＋AI攻撃パターンの多様化**（2026-07-11）：
  - **超必殺技**：`Fighter.superGauge`（蓄積のみで未使用だった）を実際に消費して発動する「必殺技」を追加。入力は弱攻撃＋強攻撃の同時押し（`INPUT_BUFFER`のバッファ猶予内でのほぼ同時＝`hasBuffered('weak') && hasBuffered('strong')`、Zキー+Xキーを完璧に同一フレームで押す必要はない）、ゲージ満タン(`superGauge>=100`)時のみ発動。新しい`FighterState='super'`を追加し、既存の攻撃ステート同様`processInput()`の早期returnリストで動作中は割り込み不可に（`updateFacing()`も除外）。スプライトは専用グラフィックが無いため`attack_strong`のポーズ画像をそのまま流用しつつ、`redrawSprite()`で金色ティントをかけて差別化。フレームデータは`SUPER_STARTUP=10`/`SUPER_ACTIVE_FRAMES=6`/`SUPER_RECOVERY=26`（`config/constants.ts`）でギア非依存の固定値（ギア5のdamageMul2.2倍と重ねてダメージが跳ね上がりすぎるのを避けるため、意図的にダメージ`SUPER_DAMAGE=260`もギア倍率をかけない据え置き）。ガードされた場合は`BattleScene.checkHit()`で`defender.guardGauge`を発動時に強制的に0にしてから`takeDamage()`を呼ぶことで、既存のガードクラッシュ計算（`dmg=round(amount*0.5)`）にそのまま乗せる形にした（新しい分岐を増やさず既存ロジックを再利用）＝**超必殺技は必ずガードを崩す**。ノックバック・ヒットストップ・画面振動も通常攻撃よりさらに大きく（`KNOCKBACK_SUPER_BONUS`等）、発動した瞬間（当たる前）に`Fighter.superJustActivated`という一度きりのフラグを立て、`BattleScene.update()`側でこれを検知して①`BattleHUD.showSuperPopup()`で必殺技名のコールアウト表示、②`camera.flash()`で軽く画面フラッシュ、③専用SE（`AudioManager`にオシレーターベースの`'super'`プリセットを追加、新規音声アセット不要）を再生——SF2的な「発動演出は当たる前に来る」感触を狙った。必殺技名は`config/parts.ts`の`SUPER_MOVE_NAME`でメカタイプ（速度/パワー/防御/バランス）ごとに変化：速度=「オーバーシフトダッシュ」、パワー=「トルクブレイカー」、防御=「リジッドカウンター」、バランス=「ギアマックス」。HUDのゲージバーも満タン時に明滅させて「撃てる」ことを視覚的に知らせる（`BattleHUD.update()`、`Math.sin(scene.time.now/120)`ベース）。ヘッドレスブラウザで、①ゲージ不足時は発動せず通常攻撃にフォールバックすること、②ゲージ満タン+ガード中の相手に対してガードクラッシュ込みの期待ダメージ（1000→870）とゲージ0リセットを確認済み。
  - **超必殺技のタイプ別個性づけ**（2026-07-11追記）：当初は「数値は共通、名前だけ変える」方式で導入したが、続けて各タイプに小さな固有トレイトを1つずつ追加（複雑化を避けるため、あくまで「1タイプ1特性」に絞った）。速度型＝発動が3フレーム速い（`Fighter.getSuperStartup()`、`startSuper()`/`tickState()`の両方でこのメソッド経由に統一し数値のズレを防止）＋ノックバック大（`BattleScene.checkHit()`の`KNOCKBACK_SPEED_SUPER_BONUS`）。パワー型＝ダメージ+15%（`getSuperDamage()`、`SUPER_DAMAGE*1.15`）。防御型＝発動窓が閉じた瞬間（＝本来ならまるごと無防備な硬直の入り口）に8フレームの無敵を付与（`tickState()`の`super`分岐、`elapsed === startup + SUPER_ACTIVE_FRAMES + 1`の1フレームだけで判定し多重付与を防止）——反撃を受け流す「タフさ」を体現。バランス型は完全据え置き（トレイトなし）。ヘッドレスブラウザで、①ガンロック(パワー)のダメージが1000→701（`round(260*1.15)=299`と一致）、②ウィズル(速度)がガンロックよりヒットが3フレーム早いこと、③アイギス(防御)の`invuln`が硬直突入の瞬間に0から8へ跳ね上がること、④ハジメ(バランス)が旧来通りの基準値（1000→740、発動1フレーム目でヒット）のままであること、の4点を個別に確認済み。既存の全回帰テストも再実行し影響なしを確認。
  - **AI攻撃パターンの多様化**（`AIController.sonicaAI()`、kakashi以外の全キャラが暫定流用しているプロファイル）：従来は「弱攻撃3連打→少し待つ」の固定パターンのみで、ギアも`fighter.gear = dist>80?1:2`という直接代入で常にGL1-2に固定、シフトのミニゲーム（パーフェクトシフト演出・シフト中の無防備）を一切経験しないためヒート/オーバーヒートの仕組みもCPU側では実質機能していなかった。今回、直接代入をやめて**本物のシフト入力**（`input.gearUp`/`gearDown`を返し、プレイヤーと同じ`Fighter.processInput()`の経路を通す）に変更：安全な距離(dist>90)で確率的にギアを上げ（上限は`escalateIfLosing()`のstageと連動、stage0はGL3まで・stage2でGL5まで＝負けが込むほど大技を仕込んでくる）、ヒートが70を超えたら自発的にギアを下げる。コンボの締めくくりで一定確率（stageで上昇）強攻撃を混ぜるように変更し弱攻撃だけの単調さを解消。ゲージが満タン(`superGauge>=100`)かつ間合い内なら弱+強の同時押し入力を返して超必殺技を撃つ。おまけで接近中に低確率のジャンプ（見た目のバリエーションのみ、専用の対空/めくり判定は未実装なので純粋な変化球）も追加。ヘッドレスブラウザで150ステップのランダム操作＋強制エスカレーションを回し、`shift`/`super`/`attack_strong`等の全状態がクラッシュなく遷移することを確認済み。

## UI/フォントとキャラサイズ（16bit風グラフィック化対応）

ユーザーから「文字がガビガビ（ジャギーが目立つ）」「PCフルスクリーン再生を考慮した綺麗な16bit(GBA/SNES)風にしてほしい」「ボスキャラはもっと大きく見せたい」との指摘があり対応済み。

### フォント刷新
- 原因：ゲーム内部解像度は384x216固定（`GAME_WIDTH`/`GAME_HEIGHT`）で、`pixelArt:true`によりcanvas全体がニアレストネイバーで4〜5倍に拡大される。以前は`sans-serif`等の一般的なアンチエイリアス済みベクターフォントを7〜9pxという極小サイズで描画していたため、荒いアンチエイリアスの縁がそのまま拡大され「ガビガビ」に見えていた。
- 対応：**PixelMplus10**（M+ BITMAP FONTSベースの日本語対応ピクセルフォント、OFL/M+ FONT LICENSE、`itouhiro/PixelMplus`）を採用。ひらがな/カタカナ/JIS第2水準漢字までカバーし、10x10ドットグリッドを基準に設計されているため小サイズでも滲まず読みやすい。
- フォントファイルは `public/fonts/PixelMplus10-{Regular,Bold}.ttf` に配置、`src/style.css` に `@font-face` 定義。`src/config/constants.ts` の `PIXEL_FONT = 'PixelMplus10'` を全テキストの `fontFamily` に使用。
- **フォントサイズは必ずPixelMplus10のネイティブグリッド(10px)の倍数にすること**（10, 20, 30...）。7px/12px/14px等の中途半端なサイズはグリフを非整数倍に再スケールすることになり、ビットマップ的な書体でも滲みが再発する。
- **フォント読み込みタイミングに注意**：`document.fonts.load()`でフォントの読み込み完了を待たずにPhaser Textオブジェクトを生成すると、フォールバックフォントで一度描画されてしまい、後からフォントが読み込まれても再描画されない（Phaser既知の挙動）。`src/scenes/BootScene.ts`の`create()`で`Promise.all([document.fonts.load(...), ...])`を待ってから`TitleScene`に遷移することで回避。

### キャラクターサイズのメリハリ
- 以前は `SPRITE_TARGET_HEIGHT` が全キャラ共通の単一定数（62px）で、`SPRITE_IDLE_SOURCE_HEIGHT`との比率だけでスケールを決めていたため、**元画像の見切り方次第でボスキャラの方が主人公より小さく表示される**という不自然な逆転が発生していた（ソフィス・レギオン等）。
- 対応：`SPRITE_TARGET_HEIGHT` を `Record<FighterId, number>` に変更（`src/config/constants.ts`）。主人公・ライバル格（hajime/wizel/ganrock/aegis/drift/theorion/kakashi）は62pxで統一、**最終ボス級（omeganova, sophislegion）は82px**にして「一回り大きいボス」感を明示的に演出。新しいボスキャラを追加する際もこのパターン（通常キャラより2〜3割高めの値）を踏襲すること。
- 注意点：ヒットボックスのサイズ（`Fighter.ts`の`reach`/`height`等）はスプライトの見た目スケールと連動しておらず固定値なので、キャラを大きくしてもヒットボックスは変わらない。今回は見た目の変更のみで許容範囲としたが、判定と見た目の乖離が気になる場合は要検討。

## スプライト素材（重要・再現性のためのノウハウ）

### 2つの生成アプローチ
- **アプローチA（手続き合成）**：`src/graphics/spriteParts.ts` + `src/graphics/SpriteFactory.ts`。パーツを2次元配列（パレットindex）で定義し、ポーズごとのオフセットで合成、Phaserのcanvasテクスチャとして実行時生成。**カカシくんのみ**この方式（外部素材が存在しない練習用ダミーのため）。他キャラへの拡張はパーツ配列とポーズ設定を追加するだけで可能。
- **アプローチB（AI画像生成）**：`docs/sprite_generation_prompts.md` に完成プロンプトあり。ハジメ・ウィズル・ガンロック・アイギス・ドリフト・テオリオン・オメガノヴァはこの方式。生成後、Pythonスクリプトで自動切り出し・透過処理・ゲーム組み込みまで行う（下記参照）。

### 均等グリッド画像が来た場合の切り出し方（テオリオンで初めて遭遇）
生成AIが黒い罫線で仕切った綺麗なグリッドを返すことがある。この場合 `scipy.ndimage` の connected-component検出より、**黒い罫線の座標を直接検出する方が速くて正確**：
```python
dark = (arr.sum(axis=2) < 60)
col_dark_frac = dark.mean(axis=0)  # 列ごとの黒率
row_dark_frac = dark.mean(axis=1)  # 行ごとの黒率
# 0.5を超える列/行がグリッド罫線
```

`public/sprites/sprite1.png`〜`sprite8.png` と `char_icons.PNG` は初期のAI生成参考画像（背景が真の透過でない・サイズ不揃い等の欠陥があり直接使用不可）。あくまで絵柄参考。

### 切り出しパイプラインで踏んだ落とし穴（次にやる人向け）
1. **背景は必ず単色グリーン(#00FF00相当)で生成させる**。白背景やチェッカー柄は自動判定が事故る（アプローチBのプロンプトに反映済み）。
2. **穴埋め処理のサイズ閾値が命**：白い装甲パーツを保つために「小さい穴だけ埋める」処理が必要だが、閾値を大きくしすぎると脚の間の隙間なども誤って塗り潰す。`MAX_HOLE_AREA=8`程度が安全。
3. **グリーンスピル（緑にじみ）除去が必要**：輪郭のアンチエイリアス部分が背景の緑と混ざり、汚い緑の縁になる。`G > max(R,B)*1.5+5` のようなG値クランプで補正（ただし**ドリフトのようにキャラ自身が緑系カラーの場合はg>150等の条件を追加しないとキャラの色まで壊れる**ので要注意）。
4. **キャラごとにidle基準の高さをスケール基準に使う**（`src/config/constants.ts` の `SPRITE_IDLE_SOURCE_HEIGHT`）。新しい元画像に差し替えたら必ずこの値を更新すること。
5. AI生成物は「顔がカメラ目線になる」「攻撃が体重移動なしの棒立ちになる」という癖がある。プロンプトの明示的な指示（`docs/sprite_generation_prompts.md` 参照）でほぼ解消したが、生成結果は毎回目視チェックすること。
6. 1枚の生成画像に約20〜34ポーズ入っていることが多い。グリッドが不揃いな場合は `scipy.ndimage` の connected-component検出＋手動マッピング、綺麗な罫線グリッドの場合は黒線座標検出（テオリオンの例、下記参照）で対応。どちらも使い捨てスクリプトなのでリポジトリには残していない。
7. **ブラウザプレビューで `window.game` のテクスチャが `__MISSING` になる／`p2`が`undefined`になる不具合が時々起きる**が、これはコードのバグではなく、プレビューツールのタブが複数回リロードされた際に `window.game` が古いPhaser.Gameインスタンスを指したままになることが原因。`preview_stop`→`preview_start`でサーバーを完全に再起動し、`document.querySelectorAll('canvas').length === 1`を確認してからテストすること。
8. **生成画像にラベル文字（"Walk Cycle"等）が焼き込まれている場合がある**（オメガノヴァで初めて遭遇）。この場合グリッド検出も連結成分検出も使えないため、各ポーズのbboxを目視で手動特定するしかない。また、bboxがパネル境界に近すぎると黒枠アーティファクトが混入するので余白を広めに取ること。加えて `keep_largest_components(mask, keep_n=2, min_frac=0.1)` フィルタ（上位2つの連結成分のみ残す）を切り出しパイプラインに追加すると、はみ出したラベル文字の破片を自動除去しつつ、正当な副成分（ナックダウンの斬撃エフェクト等）は保持できる。
9. **プレビューでポーズを手動確認する際は `scene.scene.pause()` してから `fighter.state` を書き換えて `fighter.redraw()` を呼ぶこと**。ポーズは `Fighter.currentPose` ではなく `Fighter.state`（AI/物理ループが毎フレーム上書きする）から算出されるため、pauseせずに書き換えるとスクリーンショット時には次フレームの更新で元に戻ってしまう。
10. **同一キャラのポーズ間で左右の向きが食い違うことがある**（ソフィス・レギオンで初めて遭遇：`jump`と`attack_weak`の候補コマだけ元画像内で左向きに描かれており、他の全ポーズ＝右向きというプロジェクト標準と食い違っていた）。コンテナ全体を1回だけ反転する仕組み（`facing`）なので、ポーズごとに元絵の向きがバラバラだと一部のポーズだけ攻撃方向が逆になる。切り出し前に全ポーズの向きを目視で揃え、逆向きのものは`export_pose(..., flip=True)`でそのポーズだけ個別に左右反転すること。
11. **黒線座標検出（グリッド方式）で切り出すと、グリッドの罫線そのものが前景として混入し、スプライトの縁に薄い黒枠が焼き付くことがある**（テオリオンの全12ポーズで発生、ユーザー指摘で発覚）。`bg_mask_green`は緑以外は全て前景として拾うため、黒い罫線もキャラと同じ「前景」判定になってしまうのが原因。対策：クロップ後、画像端から数px以内にほぼ全幅/全高にわたる黒不透明ピクセルの行/列がないか機械的にチェックし（`row_counts > w*0.5`かつ端からの距離が小さい等）、あれば該当行/列のalphaを0にして除去する。今後グリッド線検出方式を使う際は、切り出し時点でbboxをグリッド線から十分離す（オメガノヴァのパネル境界と同様の考え方）か、事後にこのチェックをかけること。
12. **グリーンスピル抑制の閾値`G > max(R,B)*1.5+5`は、キャラの装甲色自体がシアン/青寄りで元々Bが高い場合、十分に効かないことがある**（ソフィス・レギオンで発生：シルバー/白基調で緑要素が皆無のキャラなのに、輪郭やエフェクト部分に緑の縁取りが目立って残った）。原因はBがすでに高いため`cap=max(R,B)`が大きくなり、1.5倍の閾値を超えにくくなること。**キャラの意匠に緑色が全く含まれない場合は、より積極的な絶対値マージン方式`G > max(R,B) + 12`に切り替えて問題ない**（ドリフトのような緑系キャラでは色を壊すリスクがあるため、そちらは従来の倍率方式か`g>150`ガード付きのままにすること）。修正の要否は`(g>r+30)&(g>b+30)&(g>80)`で緑ピクセル数を数え、数百px単位で出ていれば要修正、数十px以下の孤立点なら実害なし（62px高の実サイズでは視認不可）と判断してよい。
13. **「P2側で待機ポーズが右向きに見える」という指摘は、コンテナ全体のミラーリング（`facing`によるcontainer.scaleX反転）自体は数学的に正しく機能していても起こりうる**（ウィズル/アイギス/ドリフト/テオリオン/ソフィス・レギオンで発生、ガンロック/オメガノヴァでは発生せず）。原因はidleポーズの元絵がハジメのような明確な横向きプロファイルではなく、カメラ目線に近い正面寄りのボクサー構えで描かれているため、反転してもシルエットの左右差が乏しく「反転した実感」が視覚的に伝わりにくいこと（ガンロック/オメガノヴァのidleは非対称な構えでこの問題が出なかった）。**対策：該当キャラの`idle.png`のみを`Image.FLIP_LEFT_RIGHT`で反転して保存し直す**（`public/sprites/wizel/idle.png`等）。他のポーズ（attack/walk等）は反転しないこと——全ポーズを反転するとcontainerの反転と二重に打ち消し合って攻撃時に正しく向いていた向きまで崩れてしまう。この対処はidle⇔attack遷移時にわずかな見た目の切り替わりを生む（許容範囲とユーザーが判断して採用）。新キャラのidleが同様に正面寄りの構えで「向きが分かりにくい」と指摘された場合はこのパターンを踏襲してよい。

### 新キャラを追加する手順（アプローチB）
1. `docs/sprite_generation_prompts.md` のテンプレートを使ってプロンプトを書く（キャラの見た目情報は仕様書§4.2参照）
2. 生成された画像を `public/sprites/<name>_v1.png` として配置
3. Pythonで緑背景を検出・切り出し・透過処理（上記の落とし穴に注意）してポーズごとのPNGを `public/sprites/<name>/*.png` に出力
4. `public/sprites/manifest.json` にサイズ情報を追記
5. `src/types/game.ts` の `FighterId`/`SpriteFighterId`、`src/config/constants.ts` の `SPRITE_FIGHTERS`・`SPRITE_IDLE_SOURCE_HEIGHT` にキャラを追加
6. `src/scenes/BattleScene.ts` の名前マッピング、`src/scenes/ModeSelectScene.ts` にモード追加
7. `npx tsc --noEmit` → `npm run build` → プレビューで実機確認（目線・攻撃の自然さ・向き・透過を必ずチェック）
