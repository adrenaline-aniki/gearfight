# GEAR FIGHT — 引き継ぎメモ

このファイルは、別環境でこのプロジェクトを開いたときにすぐ状況を把握できるようにするための引き継ぎ資料です。詳細な機能仕様は `docs/仕様書.md`、スプライト生成の詳細は `docs/sprite_generation_prompts.md` を参照してください。

## プロジェクト概要

中学技術科「機械・動力の伝達」学習用の2D対戦格闘ゲーム。`docs/仕様書.md` が元の完全仕様書だが、**以下の点で実装は仕様書と意図的に異なる**：

- 仕様書は「GAS単一HTML・fillRectベースの手続きスプライト」を指定しているが、実装は **Vite + TypeScript + Phaser 3 + Firebase**（デプロイはFirebase Hosting）。理由：既にこの構成で実装が進んでいたため、ゲームプレイ本質（ギア/ヒート/パーフェクトシフト等）に影響しないGAS制約は放棄した。
- スプライトは fillRect ではなく、**実画像（AI生成 + 手続き合成のハイブリッド）** を使用。詳細は下記「スプライト素材」を参照。
- それ以外（ギアシステム、ヒート、パーフェクトシフト、THEORY BONUS、クイズ、AIプロファイル等）は仕様書の記述に準拠する方針。

## UI/フォントとキャラサイズ（16bit風グラフィック化対応）

ユーザーから「文字がガビガビ（ジャギーが目立つ）」「PCフルスクリーン再生を考慮した綺麗な16bit(GBA/SNES)風にしてほしい」「ボスキャラはもっと大きく見せたい」との指摘があり対応済み。

### フォント刷新
- 原因：ゲーム内部解像度は384x216固定（`GAME_WIDTH`/`GAME_HEIGHT`）で、`pixelArt:true`によりcanvas全体がニアレストネイバーで4〜5倍に拡大される。以前は`sans-serif`等の一般的なアンチエイリアス済みベクターフォントを7〜9pxという極小サイズで描画していたため、荒いアンチエイリアスの縁がそのまま拡大され「ガビガビ」に見えていた。
- 対応：**PixelMplus10**（M+ BITMAP FONTSベースの日本語対応ピクセルフォント、OFL/M+ FONT LICENSE、`itouhiro/PixelMplus`）を採用。ひらがな/カタカナ/JIS第2水準漢字までカバーし、10x10ドットグリッドを基準に設計されているため小サイズでも滲まず読みやすい。
- フォントファイルは `public/fonts/PixelMplus10-{Regular,Bold}.ttf` に配置、`src/style.css` に `@font-face` 定義。`src/config/constants.ts` の `PIXEL_FONT = 'PixelMplus10'` を全テキストの `fontFamily` に使用。
- **フォントサイズは必ずPixelMplus10のネイティブグリッド(10px)の倍数にすること**（10, 20, 30...）。7px/12px/14px等の中途半端なサイズはグリフを非整数倍に再スケールすることになり、ビットマップ的な書体でも滲みが再発する。
- **フォント読み込みタイミングに注意**：`document.fonts.load()`でフォントの読み込み完了を待たずにPhaser Textオブジェクトを生成すると、フォールバックフォントで一度描画されてしまい、後からフォントが読み込まれても再描画されない（Phaser既知の挙動）。`src/scenes/BootScene.ts`の`create()`で`Promise.all([document.fonts.load(...), ...])`を待ってから`TitleScene`に遷移することで回避。

### キャラクターサイズのメリハリ
- 以前は `SPRITE_TARGET_HEIGHT` が全キャラ共通の単一定数（62px）で、`SPRITE_IDLE_SOURCE_HEIGHT`との比率だけでスケールを決めていたため、**元画像の見切り方次第でボスキャラの方が主人公より小さく表示される**という不自然な逆転が発生していた（ソフィス・レギオン等）。
- 対応：`SPRITE_TARGET_HEIGHT` を `Record<FighterId, number>` に変更（`src/config/constants.ts`）。主人公・ライバル格（hajime/wizel/ganrock/aegis/drift/theorion/kakashi）は62pxで統一、**最終ボス級（omeganova, sophislegion）は82px**にして「一回り大きいボス」感を明示的に演出。新しいボスキャラを追加する際もこのパターン（通常キャラより2〜3割高めの値）を踏襲すること。
- 注意点：ヒットボックスのサイズ（`Fighter.ts`の`reach`/`height`等）はスプライトの見た目スケールと連動しておらず固定値なので、キャラを大きくしてもヒットボックスは変わらない。今回は見た目の変更のみで許容範囲としたが、判定と見た目の乖離が気になる場合は要検討。

## スプライト素材（重要・再現性のためのノウハウ）

### 2つの生成アプローチ
- **アプローチA（手続き合成）**：`src/graphics/spriteParts.ts` + `src/graphics/SpriteFactory.ts`。パーツを2次元配列（パレットindex）で定義し、ポーズごとのオフセットで合成、Phaserのcanvasテクスチャとして実行時生成。**カカシくんのみ**この方式（外部素材が存在しない練習用ダミーのため）。他キャラへの拡張はパーツ配列とポーズ設定を追加するだけで可能。
- **アプローチB（AI画像生成）**：`docs/sprite_generation_prompts.md` に完成プロンプトあり。ハジメ・ウィズル・ガンロック・アイギス・ドリフト・テオリオン・オメガノヴァはこの方式。生成後、Pythonスクリプトで自動切り出し・透過処理・ゲーム組み込みまで行う（下記参照）。

### 均等グリッド画像が来た場合の切り出し方（テオリオンで初めて遭遇）
生成AIが黒い罫線で仕切った綺麗なグリッドを返すことがある。この場合 `scipy.ndimage` の connected-component検出より、**黒い罫線の座標を直接検出する方が速くて正確**：
```python
dark = (arr.sum(axis=2) < 60)
col_dark_frac = dark.mean(axis=0)  # 列ごとの黒率
row_dark_frac = dark.mean(axis=1)  # 行ごとの黒率
# 0.5を超える列/行がグリッド罫線
```

`public/sprites/sprite1.png`〜`sprite8.png` と `char_icons.PNG` は初期のAI生成参考画像（背景が真の透過でない・サイズ不揃い等の欠陥があり直接使用不可）。あくまで絵柄参考。

### 切り出しパイプラインで踏んだ落とし穴（次にやる人向け）
1. **背景は必ず単色グリーン(#00FF00相当)で生成させる**。白背景やチェッカー柄は自動判定が事故る（アプローチBのプロンプトに反映済み）。
2. **穴埋め処理のサイズ閾値が命**：白い装甲パーツを保つために「小さい穴だけ埋める」処理が必要だが、閾値を大きくしすぎると脚の間の隙間なども誤って塗り潰す。`MAX_HOLE_AREA=8`程度が安全。
3. **グリーンスピル（緑にじみ）除去が必要**：輪郭のアンチエイリアス部分が背景の緑と混ざり、汚い緑の縁になる。`G > max(R,B)*1.5+5` のようなG値クランプで補正（ただし**ドリフトのようにキャラ自身が緑系カラーの場合はg>150等の条件を追加しないとキャラの色まで壊れる**ので要注意）。
4. **キャラごとにidle基準の高さをスケール基準に使う**（`src/config/constants.ts` の `SPRITE_IDLE_SOURCE_HEIGHT`）。新しい元画像に差し替えたら必ずこの値を更新すること。
5. AI生成物は「顔がカメラ目線になる」「攻撃が体重移動なしの棒立ちになる」という癖がある。プロンプトの明示的な指示（`docs/sprite_generation_prompts.md` 参照）でほぼ解消したが、生成結果は毎回目視チェックすること。
6. 1枚の生成画像に約20〜34ポーズ入っていることが多い。グリッドが不揃いな場合は `scipy.ndimage` の connected-component検出＋手動マッピング、綺麗な罫線グリッドの場合は黒線座標検出（テオリオンの例、下記参照）で対応。どちらも使い捨てスクリプトなのでリポジトリには残していない。
7. **ブラウザプレビューで `window.game` のテクスチャが `__MISSING` になる／`p2`が`undefined`になる不具合が時々起きる**が、これはコードのバグではなく、プレビューツールのタブが複数回リロードされた際に `window.game` が古いPhaser.Gameインスタンスを指したままになることが原因。`preview_stop`→`preview_start`でサーバーを完全に再起動し、`document.querySelectorAll('canvas').length === 1`を確認してからテストすること。
8. **生成画像にラベル文字（"Walk Cycle"等）が焼き込まれている場合がある**（オメガノヴァで初めて遭遇）。この場合グリッド検出も連結成分検出も使えないため、各ポーズのbboxを目視で手動特定するしかない。また、bboxがパネル境界に近すぎると黒枠アーティファクトが混入するので余白を広めに取ること。加えて `keep_largest_components(mask, keep_n=2, min_frac=0.1)` フィルタ（上位2つの連結成分のみ残す）を切り出しパイプラインに追加すると、はみ出したラベル文字の破片を自動除去しつつ、正当な副成分（ナックダウンの斬撃エフェクト等）は保持できる。
9. **プレビューでポーズを手動確認する際は `scene.scene.pause()` してから `fighter.state` を書き換えて `fighter.redraw()` を呼ぶこと**。ポーズは `Fighter.currentPose` ではなく `Fighter.state`（AI/物理ループが毎フレーム上書きする）から算出されるため、pauseせずに書き換えるとスクリーンショット時には次フレームの更新で元に戻ってしまう。
10. **同一キャラのポーズ間で左右の向きが食い違うことがある**（ソフィス・レギオンで初めて遭遇：`jump`と`attack_weak`の候補コマだけ元画像内で左向きに描かれており、他の全ポーズ＝右向きというプロジェクト標準と食い違っていた）。コンテナ全体を1回だけ反転する仕組み（`facing`）なので、ポーズごとに元絵の向きがバラバラだと一部のポーズだけ攻撃方向が逆になる。切り出し前に全ポーズの向きを目視で揃え、逆向きのものは`export_pose(..., flip=True)`でそのポーズだけ個別に左右反転すること。

### 新キャラを追加する手順（アプローチB）
1. `docs/sprite_generation_prompts.md` のテンプレートを使ってプロンプトを書く（キャラの見た目情報は仕様書§4.2参照）
2. 生成された画像を `public/sprites/<name>_v1.png` として配置
3. Pythonで緑背景を検出・切り出し・透過処理（上記の落とし穴に注意）してポーズごとのPNGを `public/sprites/<name>/*.png` に出力
4. `public/sprites/manifest.json` にサイズ情報を追記
5. `src/types/game.ts` の `FighterId`/`SpriteFighterId`、`src/config/constants.ts` の `SPRITE_FIGHTERS`・`SPRITE_IDLE_SOURCE_HEIGHT` にキャラを追加
6. `src/scenes/BattleScene.ts` の名前マッピング、`src/scenes/ModeSelectScene.ts` にモード追加
7. `npx tsc --noEmit` → `npm run build` → プレビューで実機確認（目線・攻撃の自然さ・向き・透過を必ずチェック）
